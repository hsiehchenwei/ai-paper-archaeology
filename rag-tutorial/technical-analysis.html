💡 本頁面目標

本頁面深入探討 RAG 的技術層面：從檢索器如何運作、生成器如何整合檢索結果， 到 RAG 面臨的技術挑戰與解決方案。適合想要實作 RAG 系統或理解技術細節的讀者。

🏗️ RAG 架構運作原理

1. 核心架構：檢索器 + 生成器

RAG 系統由兩個核心組件組成：檢索器（Retriever）和生成器（Generator）。 這兩個組件協同工作，實現「先查資料再回答」的流程。

📊 RAG 架構流程：Query → Retrieve → Augment → Generate

🔍 檢索器（Retriever）

職責：根據使用者查詢，從知識庫中找出最相關的文檔片段。

  • 將查詢轉換為向量表示
  • 在向量資料庫中進行相似度搜尋
  • 返回 Top-K 個最相關的文檔

✍️ 生成器（Generator）

職責：基於檢索到的文檔和使用者查詢，生成最終回答。

  • 接收查詢 + 檢索文檔
  • 理解上下文關係
  • 生成自然語言回答

2. 完整流程解析

  1. 步驟 1：查詢編碼（Query Encoding）
    使用者輸入「台灣第一座核電廠是誰設計的？」，檢索器將此查詢轉換為向量表示。
  2. 步驟 2：向量檢索（Vector Retrieval）
    在向量資料庫中搜尋與查詢向量最相似的文檔向量，通常使用餘弦相似度或點積。
  3. 步驟 3：文檔排序（Re-ranking，可選）
    使用更精細的模型（如 Cross-Encoder）對檢索結果重新排序，提升準確度。
  4. 步驟 4：上下文構建（Context Construction）
    將檢索到的文檔片段與原始查詢組合成 prompt，例如：「基於以下文檔回答問題：...」
  5. 步驟 5：生成回答（Answer Generation）
    語言模型基於上下文生成最終回答，並可選擇性地提供來源引用。

📈 檢索技術演進：從 BM25 到 Hybrid

🔍 檢索技術的三個時代

檢索技術從傳統的關鍵字匹配，演進到語義理解，再到混合式檢索。 每個階段都有其優勢與限制。

1. Sparse Retrieval (稀疏檢索)

代表：BM25、TF-IDF

工作原理

  • 基於關鍵字匹配，計算查詢與文檔的詞彙重疊度
  • BM25 考慮詞頻、逆文檔頻率、文檔長度正規化
  • 速度快，但無法理解語義
優點：速度快、可解釋性高、無需訓練
缺點：無法處理同義詞、語義理解弱

2. Dense Retrieval (密集檢索)

代表：DPR、Sentence-BERT、OpenAI Embeddings

工作原理

  • 使用神經網路將查詢和文檔編碼為密集向量
  • 透過向量相似度（餘弦相似度）進行檢索
  • 能理解語義，處理同義詞和語義相關性
優點：語義理解強、能處理同義詞
缺點：需要訓練、計算成本高、可能遺漏關鍵字

3. Hybrid Retrieval (混合檢索)

代表：BM25 + Dense、ColBERT、SPLADE

工作原理

  • 結合 Sparse 和 Dense 檢索的優勢
  • 使用加權分數融合兩種檢索結果
  • 例如：score = α × BM25_score + (1-α) × Dense_score
優點：結合兩者優勢，準確度最高
缺點：計算成本最高、需要調參

⚠️ RAG 的技術缺陷與挑戰

🚨 為什麼 RAG 不是萬能解藥？

儘管 RAG 解決了 LLM 的知識時效性問題，但它本身也面臨許多技術挑戰。 理解這些缺陷，有助於在實作時做出正確的技術選擇。

1. 檢索品質問題

問題：檢索器可能返回不相關的文檔，導致生成器基於錯誤資訊產生答案。

具體表現

  • 語義漂移：查詢「蘋果公司」卻檢索到「蘋果水果」相關文檔
  • 關鍵字匹配失敗：查詢「AI 模型」但文檔使用「機器學習模型」
  • 上下文缺失：檢索到片段但缺少必要的上下文資訊
解決方案：Query Rewriting、Re-ranking、Hybrid Retrieval、Self-RAG 反思機制

2. 上下文長度限制

問題：檢索到的文檔片段需要放入 LLM 的 context window，但 context 長度有限。

具體表現

  • 資訊截斷：只能放入 Top-K 個文檔，可能遺漏關鍵資訊
  • 長文檔處理：單一文檔過長時，需要分塊但可能破壞語義完整性
  • 多輪對話：對話歷史 + 檢索文檔可能超過 context 限制
解決方案：智能分塊（Semantic Chunking）、文檔摘要、Long Context LLM、階層式檢索（HiRAG）

3. 組合問題失敗

問題：需要整合多個文檔資訊才能回答的問題，RAG 表現不佳。

具體表現

  • 跨文檔推理：「比較 A 和 B 的差異」需要檢索兩個文檔並整合
  • 多步驟問題：「先做 X 再做 Y 的結果是什麼？」需要多輪檢索
  • 全局資訊：需要理解整個知識庫的結構（GraphRAG 試圖解決）
解決方案：GraphRAG（知識圖譜）、Multi-Agent RAG、Iterative Retrieval、Agentic RAG

4. 無法根除幻覺

問題：即使檢索到正確資訊，LLM 仍可能產生錯誤答案或無法拒絕錯誤上下文。

具體表現

  • 錯誤上下文接受：RefusalBench 研究顯示，RAG 系統在錯誤上下文時無法拒絕
  • 資訊混淆：將多個文檔的資訊錯誤組合
  • 過度推斷：基於檢索資訊做出超出範圍的推斷
解決方案：Self-RAG 反思機制、Fact-checking、來源驗證、Prompt Engineering

🗄️ 向量資料庫比較

💾 選擇合適的向量資料庫

向量資料庫是 RAG 系統的基礎設施。不同的資料庫有不同的特性，選擇合適的資料庫對系統效能至關重要。

資料庫 類型 擴展性 易用性 成本 適用場景
Pinecone 雲端 SaaS ⭐⭐⭐⭐⭐ ⭐⭐⭐⭐⭐ $$$ 快速原型、中小型專案
Weaviate 開源 / 雲端 ⭐⭐⭐⭐ ⭐⭐⭐⭐ $$ 中大型專案、自託管
Qdrant 開源 ⭐⭐⭐⭐ ⭐⭐⭐ $ 自託管、成本敏感
Chroma 開源 ⭐⭐⭐ ⭐⭐⭐⭐⭐ $ 開發測試、小型專案
Milvus 開源 ⭐⭐⭐⭐⭐ ⭐⭐ $ 超大規模、企業級
pgvector PostgreSQL 擴展 ⭐⭐⭐ ⭐⭐⭐ $ 已有 PostgreSQL 基礎設施
