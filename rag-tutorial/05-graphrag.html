A Graph RAG Approach to Query-Focused Summarization

Darren Edge, Ha Trinh, Newman Cheng, et al.

🌐 解決傳統 RAG 的「局部問題」限制

💡 一句話總結

GraphRAG 整合知識圖譜，突破傳統 RAG 只能回答「局部問題」的限制。 透過 Community Detection 和全局摘要，實現跨文檔推理與全局視角回答。

✅ Local Questions
(局部問題)

「誰發明了電話？」
「巴黎是哪裡的首都？」
傳統 RAG 擅長

❌ Global Questions
(全局問題)

「這個資料集的主要主題是什麼？」
「這些事件之間有什麼關聯？」
傳統 RAG 無法回答

GraphRAG 的目標：讓 RAG 也能回答全局問題！

🔧 GraphRAG 架構

📊 GraphRAG 三階段流程：知識圖譜建構 → 社群偵測 → 查詢處理

💡 論文原始圖表

GraphRAG 論文中的 Figure 1 展示了完整的系統架構：

  • Source Documents：原始文檔作為輸入
  • Element Instances：提取實體、關係和主張（claims）
  • Element Summaries：為每個元素生成摘要
  • Graph Communities：基於圖譜的社群偵測（Leiden 演算法）
  • Community Summaries：為每個社群生成層次化摘要
  • Community Answers：使用社群摘要回答全局問題

📄 請參考 論文 PDF 第 2 頁 Figure 1

三階段 Pipeline

1️⃣ Knowledge Graph 建構

  • 從文檔中提取實體（人物、地點、組織等）
  • 識別實體之間的關係
  • 建立知識圖譜（Entity → Relation → Entity）

2️⃣ Community Detection

  • 在知識圖譜中找出相關實體的「社群」
  • 每個社群代表一個主題或概念
  • 為每個社群生成摘要

3️⃣ Query Processing

  • Local 查詢：直接檢索相關文檔
  • Global 查詢：使用社群摘要回答
  • 結合兩種策略提供完整答案

🎯 生活類比：圖書館 vs 知識地圖

傳統 RAG = 圖書館目錄

  • 可以找到「某本書在哪裡」
  • 但無法回答「這些書的共同主題是什麼」

GraphRAG = 知識地圖

  • 不只知道「書在哪裡」
  • 還知道「書之間的關聯」
  • 可以回答「整個圖書館的主題分布」

💡 核心技術

🕸️ Knowledge Graph 建構

GraphRAG 使用 LLM 從文檔中提取實體和關係：

  1. 實體提取：識別文檔中的人物、組織、地點等
  2. 關係識別：找出實體之間的關係（例如：A 是 B 的創辦人）
  3. 圖譜建構：建立 Entity-Relation-Entity 三元組
  4. 圖譜優化：合併重複實體，清理噪音

👥 Community Detection

在知識圖譜中找出相關實體的社群：

  • 使用圖譜演算法（如 Louvain）找出緊密相連的實體群組
  • 每個社群代表一個主題或概念
  • 為每個社群生成摘要，描述該社群的核心內容

這些社群摘要就是回答全局問題的關鍵！

🔧 論文實作細節

📝 階段一：Source Documents → Text Chunks

文檔分塊策略

  • 預設塊大小：600 tokens（論文實驗設定）
  • 塊重疊：100 tokens 重疊，確保資訊不遺漏
  • 設計權衡：
    • 較長的塊 → 減少 LLM 調用次數（降低成本）
    • 較短的塊 → 提高資訊召回率（避免早期資訊遺漏）
  • 實際案例：
    • Podcast 資料集：1669 個 600-token 塊（約 100 萬 tokens）
    • News 資料集：3197 個 600-token 塊（約 170 萬 tokens）

🔍 階段二：Text Chunks → Entities & Relationships

LLM 提取實體與關係

📌 論文範例

"NeoChip's (NC) shares surged in their first week of trading on the NewTech Exchange. However, market analysts caution that the chipmaker's public debut may not reflect trends for other technology IPOs. NeoChip, previously a private entity, was acquired by Quantum Systems in 2016."

LLM 提取結果：

  • 實體：NeoChip → "NeoChip is a publicly traded company specializing in low-power processors for wearables and IoT devices."
  • 實體：Quantum Systems → "Quantum Systems is a firm that previously owned NeoChip."
  • 關係 → "Quantum Systems owned NeoChip from 2016 until NeoChip became publicly traded."
  • 提取內容：
    • 實體（Entities）：人物、組織、地點等，包含名稱、類型、描述
    • 關係（Relationships）：實體間的關係，包含來源、目標、關係描述
    • 主張（Claims）：關於實體的重要事實陳述（日期、事件、互動等）
  • 領域適應：使用 few-shot exemplars 進行領域特定調整
    • 預設提示詞適用於一般命名實體（人物、地點、組織）
    • 專業領域（科學、醫學、法律）可使用領域特定的範例
  • Self-Reflection 技術：
    • LLM 先生成提取結果，再評估正確性、清晰度、完整性
    • 基於評估結果生成改進的響應
    • 在 HotPotQA 資料集上，600-token 塊比 2400-token 塊提取的實體數量多近兩倍

🕸️ 階段三：Entities & Relationships → Knowledge Graph

建構知識圖譜

  • 節點（Nodes）：每個實體實例成為圖中的一個節點
  • 邊（Edges）：關係實例成為圖中的邊
  • 聚合與摘要：
    • 實體描述被聚合並為每個節點生成摘要
    • 關係的數量成為邊的權重（重複關係越多，權重越大）
    • 主張（claims）也以類似方式聚合
  • 實體匹配：
    • 論文使用精確字串匹配（exact string matching）
    • 可調整為更軟的匹配方法（fuzzy matching）
    • GraphRAG 對重複實體具有韌性，因為重複實體通常會在後續步驟中被聚類
  • 抽象化摘要：提取過程本身就是一種抽象化摘要
    • 關係和主張可能不會在原文中明確陳述
    • LLM 從文本中推斷出隱含的關係

👥 階段四：Knowledge Graph → Graph Communities

階層式社群偵測

  • 演算法選擇：使用 Leiden 社群偵測演算法（Traag et al., 2019）
  • 階層式分割：
    • 遞迴地在每個偵測到的社群內偵測子社群
    • 持續分割直到達到無法再分割的葉子社群（leaf communities）
    • 每個層級提供一個互斥且完整的社群分割
  • 階層結構的優勢：
    • 支援分治法（divide-and-conquer）全局摘要
    • 不同層級可回答不同粒度的問題
    • 高層級社群提供廣泛視角，低層級社群提供詳細資訊

📄 階段五：Graph Communities → Community Summaries

生成社群摘要

🌳 葉子級社群（Leaf-level Communities）
  1. 按節點度數（node degree）優先順序排序社群邊
  2. 依序添加以下元素到 LLM context window：
    • 來源節點描述
    • 目標節點描述
    • 邊本身描述
    • 相關主張（claims）
  3. 持續添加直到達到 token 限制
🏛️ 高級社群（Higher-level Communities）
  1. 如果所有元素摘要都能放入 context window → 直接摘要所有元素
  2. 如果超過 token 限制：
    • 按元素摘要的 token 數量降序排序子社群
    • 迭代地用子社群摘要（較短）替換對應的元素摘要（較長）
    • 直到符合 context window 限制

💡 設計理念：這些摘要本身就有用，可以幫助理解資料集的全局結構和語義，即使沒有特定查詢也能用來理解語料庫。

🔎 階段六：Community Summaries → Community Answers → Global Answer

Map-Reduce 查詢處理

📦 Step 1: 準備社群摘要
  • 社群摘要被隨機打亂（random shuffle）
  • 分割成預定 token 大小的塊（chunks）
  • 目的：確保相關資訊分散在多個塊中，而不是集中在單一 context window（避免資訊遺漏）
🗺️ Step 2: Map 階段（並行生成中間答案）
  • 為每個社群摘要塊並行生成中間答案
  • LLM 同時生成：
    • 部分答案（partial answer）
    • 有用性分數（helpfulness score）：0-100 分
  • 過濾：分數為 0 的答案被過濾掉
📊 Step 3: Reduce 階段（生成全局答案）
  1. 中間答案按有用性分數降序排序
  2. 迭代地將答案添加到新的 context window
  3. 持續添加直到達到 token 限制
  4. 使用最終 context 生成返回給用戶的全局答案

🎯 階層結構的靈活性：不同層級的社群摘要可用於回答不同粒度的問題，論文評估了哪個層級在摘要詳細程度和範圍之間提供最佳平衡。

⚙️ 技術配置與效能

🔧 實驗配置
  • Context Window：
    • 社群摘要生成：8k tokens
    • 社群答案生成：8k tokens
    • 全局答案生成：8k tokens
    • 圖索引建構：600 tokens（用於實體提取）
  • 硬體環境：
    • 虛擬機器：16GB RAM
    • CPU：Intel(R) Xeon(R) Platinum 8171M @ 2.60GHz
    • LLM API：OpenAI gpt-4-turbo（2M TPM, 10k RPM）
  • 處理時間：
    • Podcast 資料集（100 萬 tokens）：圖索引建構約 281 分鐘
📈 效能優化技巧
  • Self-Reflection：提升實體提取品質，但增加 token 消耗
  • 塊大小選擇：600 tokens 在成本和召回率之間取得平衡
  • 並行處理：Map 階段可並行生成多個社群答案
  • 階層式摘要：利用社群階層結構，避免重複處理

🤖 模型使用方式：Prompt Engineering vs 微調

✅ 使用現有模型，透過 Prompt 進行多次任務

GraphRAG 並未微調模型參數，而是使用現有的 LLM（GPT-4）透過精心設計的 prompt 來完成所有任務。 這是一個純粹的 Prompt Engineering 方法，不需要訓練或調整模型權重。

🔧 使用的模型與 API
  • 模型：GPT-4（OpenAI）
  • API：OpenAI public endpoint for gpt-4-turbo
  • API 限制：2M TPM (Tokens Per Minute), 10k RPM (Requests Per Minute)
  • 模型狀態：預訓練模型，未進行微調
📝 Prompt Engineering 技術

GraphRAG 使用多種 Prompt Engineering 技術：

  • Few-shot Learning（少樣本學習）：
    • 在 prompt 中提供領域特定的範例（exemplars）
    • 讓模型透過 in-context learning 理解任務
    • 例如：提供科學、醫學、法律等領域的實體提取範例
  • Self-Reflection（自我反思）：
    • LLM 先生成答案
    • 然後評估答案的正確性、清晰度、完整性
    • 基於評估結果生成改進的響應
    • 這是一個多輪 prompt 技術，提升提取品質
  • Prompt Templates（提示詞模板）：
    • 為不同任務設計專門的 prompt 模板
    • 實體提取、關係提取、主張提取各有專用 prompt
    • 社群摘要、全局答案生成也有對應的 prompt 設計
  • Domain-specific Exemplars（領域特定範例）：
    • 預設 prompt 適用於一般命名實體（人物、地點、組織）
    • 專業領域可提供領域特定的 few-shot 範例
    • 例如：科學領域的實體類型、醫學領域的關係類型等
🔄 多次 LLM 調用流程

GraphRAG 在整個流程中進行多次 LLM API 調用：

  1. 索引建構階段（Indexing Time）：
    • 為每個文字塊調用 LLM 提取實體和關係
    • Podcast 資料集：1669 個塊 → 1669 次 LLM 調用
    • News 資料集：3197 個塊 → 3197 次 LLM 調用
    • 如果使用 Self-Reflection，每個塊需要 2-3 次調用
  2. 社群摘要生成階段：
    • 為每個社群（葉子級和高級）調用 LLM 生成摘要
    • 社群數量取決於資料集大小和社群偵測結果
    • 可並行處理多個社群摘要
  3. 查詢處理階段（Query Time）：
    • Map 階段：為每個社群摘要塊並行調用 LLM 生成部分答案
    • Reduce 階段：調用 LLM 將部分答案合併成全局答案

💡 關鍵點：所有這些調用都使用相同的預訓練 GPT-4 模型，只是透過不同的 prompt 來引導模型完成不同任務。

📊 為什麼不微調？
  • 通用性：使用現有模型可以快速應用到不同領域，只需調整 prompt 中的 few-shot 範例
  • 成本效益：微調需要大量標註資料和計算資源，而 prompt engineering 成本較低
  • 靈活性：可以針對不同任務快速調整 prompt，無需重新訓練模型
  • 可解釋性：Prompt 的設計邏輯清晰，容易理解和調整
  • 實用性：GPT-4 等大型模型已經具備強大的推理能力，透過 prompt 即可引導完成複雜任務
🎯 論文中的證據
  • 論文明確提到："The LLM is prompted to extract..."（LLM 被提示提取...）
  • "These prompts can be tailored to the domain..."（這些 prompt 可以針對領域調整...）
  • "few-shot exemplars for in-context learning"（用於上下文學習的少樣本範例）
  • "using a public OpenAI endpoint for gpt-4-turbo"（使用 OpenAI 公開端點）
  • Appendix A 和 Appendix E 詳細描述了各種 prompt 的設計

📊 應用場景

✅ GraphRAG 擅長的任務

  • 全局摘要：「這個資料集的主要主題是什麼？」
  • 關聯分析：「這些事件之間有什麼關聯？」
  • 趨勢分析：「這個領域的發展趨勢是什麼？」
  • 欺詐檢測：識別異常模式和關聯

🔄 GraphRAG vs 傳統 RAG

面向 傳統 RAG GraphRAG
知識表示 向量索引（扁平） 知識圖譜（結構化）
查詢類型 局部問題 局部 + 全局問題
推理能力 單一文檔 跨文檔推理
計算成本 較低 較高（需建構圖譜）
適用規模 小到中型資料集 大型資料集（百萬 token）

🔬 評測驗證：實驗設計與測試結果

📋 實驗概述

論文進行了兩個主要實驗，使用 Adaptive Benchmarking 方法生成測試問題， 並使用 LLM-as-a-Judge 技術進行評估。所有實驗都使用 GPT-4 作為 LLM。

❓ 問題生成方法：Adaptive Benchmarking

為什麼需要新的評估方法？

  • 傳統基準（如 HotPotQA、MultiHop-RAG）專注於事實檢索，不適合全局理解任務
  • 全局 sensemaking 問題沒有標準答案（ground truth）
  • 需要生成能代表真實使用場景的問題
🔧 三階段問題生成流程
  1. 生成用戶角色（Personas）：
    • 基於資料集描述，LLM 生成 K 個潛在用戶角色
    • 例如：科技記者、教育工作者、研究人員等
  2. 識別任務（Tasks）：
    • 為每個用戶角色識別 N 個相關任務
    • 例如：理解科技政策、整合健康議題到課程等
  3. 生成問題（Questions）：
    • 為每個（用戶，任務）組合生成 M 個全局理解問題
    • 問題要求理解整個語料庫，而非檢索特定事實

📊 實驗設定：K = M = N = 5，每個資料集生成 125 個測試問題（5 × 5 × 5）

📝 問題範例

Podcast 資料集：

  • 用戶：科技記者，任務：理解科技領袖對政策與監管的看法
  • 問題範例：
    • "哪些集數主要討論科技政策和政府監管？"
    • "來賓如何看待隱私法對科技發展的影響？"
    • "來賓是否討論創新與倫理考量之間的平衡？"

News 資料集：

  • 用戶：教育工作者，任務：將時事融入課程
  • 問題範例：
    • "哪些當前健康議題可以整合到健康教育課程中？"
    • "新聞文章如何處理預防醫學和健康概念？"
    • "基於新聞報導，可以從中了解哪些公共衛生優先事項？"

📊 評估標準：四個維度

由於全局 sensemaking 問題沒有標準答案，論文使用 Head-to-Head Comparison 方法， 讓 LLM 評估兩個系統的相對表現。

📈 Comprehensiveness
（全面性）

答案提供了多少細節來涵蓋問題的所有面向和細節？

🌈 Diversity
（多樣性）

答案在提供不同觀點和見解方面有多豐富和多樣？

💡 Empowerment
（賦能性）

答案如何幫助讀者理解並做出明智判斷？

🎯 Directness
（直接性）

答案如何具體且清晰地回答問題？（對照標準）

⚖️ 評估流程
  1. LLM 接收問題和兩個競爭系統的答案
  2. 根據特定標準比較兩個答案
  3. 判斷哪個答案更好，或標記為平手（tie）
  4. 為考慮 LLM 的隨機性，每個比較進行多次重複（replicates）
  5. 計算平均結果：獲勝 = 100 分，失敗 = 0 分，平手 = 50 分

💡 Directness 作為對照標準：由於它與 Comprehensiveness 和 Diversity 對立， 預期沒有方法能在所有四個標準上都獲勝。

🧪 對照實驗設計：六個條件

論文比較了六個條件，所有條件使用相同的 context window 大小（8k tokens）和相同的 prompt， 差異僅在於如何創建 context window 的內容。

🕸️ GraphRAG 條件（C0-C3）
  • C0：使用根層級社群摘要（數量最少）
  • C1：使用高層級社群摘要（C0 的子社群，如果存在）
  • C2：使用中層級社群摘要（C1 的子社群，如果存在）
  • C3：使用低層級社群摘要（數量最多，C2 的子社群）

💡 這些條件測試不同層級的社群摘要對答案品質的影響

📄 對照條件
  • TS（Text Summarization）：
    • 使用相同的 map-reduce 方法，但直接對源文本進行摘要
    • 源文本被打亂並分塊，而非使用社群摘要
    • 測試社群摘要是否比直接文本摘要更好
  • SS（Semantic Search / Vector RAG）：
    • 傳統向量 RAG 實作
    • 檢索語義相似的文本塊，添加到 context window 直到達到 token 限制
    • 這是 GraphRAG 的主要對照基準

📈 測試資料集

🎙️ Podcast 資料集
  • 來源：Behind the Tech with Kevin Scott 播客轉錄
  • 內容：Microsoft CTO 與科技領袖的對話
  • 規模：1669 個 600-token 塊（約 100 萬 tokens）
  • 重疊：100 tokens
  • 圖譜：8,564 個節點，20,691 條邊
📰 News 資料集
  • 來源：2013 年 9 月至 2023 年 12 月的新聞文章
  • 類別：娛樂、商業、體育、科技、健康、科學
  • 規模：3197 個 600-token 塊（約 170 萬 tokens）
  • 重疊：100 tokens
  • 圖譜：15,754 個節點，19,520 條邊

🏆 測試結果：Experiment 1

✅ GraphRAG vs Vector RAG（SS）

Comprehensiveness（全面性）：

  • Podcast 資料集：GraphRAG 獲勝率 72-83%（p < 0.001）
  • News 資料集：GraphRAG 獲勝率 72-80%（p < 0.001）
  • 所有 GraphRAG 條件（C0-C3）都顯著優於向量 RAG

Diversity（多樣性）：

  • Podcast 資料集：GraphRAG 獲勝率 75-82%（p < 0.001）
  • News 資料集：GraphRAG 獲勝率 62-71%（p < 0.01）

Directness（直接性）：

  • 向量 RAG（SS）在所有比較中產生最直接的答案
  • 這符合預期，因為 Directness 與 Comprehensiveness 和 Diversity 對立
📊 GraphRAG 不同層級比較

Comprehensiveness：

  • C2 和 C3 在兩個資料集上都顯著優於 TS（p < 0.001）
  • C1 在 News 資料集上優於 TS（p = 0.002）
  • 較低層級的社群摘要（C2, C3）提供更詳細的資訊

Empowerment（賦能性）：

  • 結果混合：GraphRAG 與向量 RAG 之間沒有明確勝者
  • GraphRAG 與 TS 之間也沒有明確勝者
  • LLM 評估顯示這可能與提供具體範例的能力有關
📈 統計分析
  • 統計檢驗：使用 Wilcoxon signed-rank test（非參數檢驗）
  • 多重比較校正：Holm-Bonferroni correction
  • 重複次數：每個比較進行 5 次評估運行
  • 顯著性水平：p < 0.001（非常顯著），p < 0.01（顯著）

🔬 Experiment 2：基於主張的驗證

為驗證 Experiment 1 的結果，論文進行了第二個實驗，使用事實主張（factual claims）作為客觀指標。

📝 主張提取方法
  • 工具：使用 Claimify（LLM-based 方法）
  • 定義：主張是「明確呈現某些可驗證事實的陳述」
  • 流程：
    1. 識別答案中包含至少一個事實主張的句子
    2. 將這些句子分解為簡單、自包含的事實主張
    3. 移除重複主張
  • 結果：從所有答案中提取了 47,075 個唯一主張，平均每個答案 31 個主張
📊 兩個指標
  • Comprehensiveness（全面性）：
    • 測量每個條件下答案中提取的主張平均數量
    • 更多主張 = 更全面
  • Diversity（多樣性）：
    • 對每個答案的主張進行聚類
    • 計算平均聚類數量
    • 使用 Scikit-learn 的 Agglomerative Clustering
    • 距離度量：1 - ROUGE-L
    • 更多聚類 = 更多樣

💡 這個實驗提供了客觀的、可量化的驗證，支持 Experiment 1 中使用 LLM-as-a-Judge 的主觀評估結果。

💡 實驗設計的創新點

  • Adaptive Benchmarking：針對特定領域動態生成評估基準，而非使用通用基準
  • LLM-as-a-Judge：使用 LLM 評估沒有標準答案的全局理解問題
  • 多維度評估：不僅評估準確性，還評估全面性、多樣性、賦能性
  • 客觀驗證：使用事實主張作為客觀指標驗證主觀評估結果
  • 公平對照：所有條件使用相同的 context window 和 prompt，只改變內容來源
